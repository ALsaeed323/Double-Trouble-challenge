Double Trouble — Write-up

Unlocking an Android app with hidden screens and secret knocks

ASCWG{Misc0nfig_Caus3_Leakag3}

An old APK sits quiet—too quiet. The rumor: "knock once… then again… and the vault wakes." Here's how we found the rhythm and opened it.

Key points

The app has two gates into a hidden screen:

  1. A password that is randomly generated at runtime (impossible to guess)
  2. A "knock" counter (repeated launches) that eventually opens a second activity

A third, cleaner path exists: start the hidden screen with an internal intent extra: from_password=true.

We used Frida to (a) simulate knocks, and (b) then directly launch the private activity with the right extra.

1) Recon

Manifest highlights

  • Package: com.hacking.test
  • Activities:
    • com.hacking.test.MainActivity (exported)
    • com.hacking.test.SecondActivity (not exported) ← the hidden room
  • No broadcast/intent entry points to SecondActivity from outside

First impressions (running the APK)

Main screen is a simple password prompt that always says "Wrong password".

2) Static analysis

MainActivity

onCreate(...) {
  setContentView(R.layout.activity_main);
  f2024v = UUID.randomUUID().toString().substring(0, 20);  // random password
  Button.onClick -> {
    if (!input.equals(f2024v)) { Toast "Wrong password"; return; }
    Intent i = new Intent(this, /* (Class) null in decomp view */);
    i.putExtra("from_password", true);
    startActivity(i);
  }
}

The "password" is different on every launch ⇒ bruteforce is pointless. The important bit is the extra from_password=true the app intends to send if the check passed.

SecondActivity

onCreate(...) {
  int threshold = new Random().nextInt(20) + 1; // 1..20
  SharedPreferences sp = getSharedPreferences("SecretPrefs", 0);
  int count = sp.getInt("launch_count", 0);

  if (!getIntent().getBooleanExtra("from_password", false)) {
    int next = count + 1;
    sp.edit().putInt("launch_count", next).apply();
    if (next < threshold) { finish(); return; } // go away until "enough knocks"
  }

  sp.edit().putInt("launch_count", 0).apply(); // reset on success
  setContentView(R.layout.activity_second);    // flag screen (mirrored)
}

Takeaway

There are three ways in:

  1. Type the random password → sets from_password=true → opens. (Not practical.)
  2. "Knock": repeatedly start SecondActivity until launch_count >= threshold. (Annoying but doable.)
  3. Directly start SecondActivity from inside the app with from_password=true. (Best.)

3) Dynamic plan

A) Simulate the knocks

Prove the mechanic by hooking MainActivity.onResume() and quickly re-launching the activity chain to bump launch_count. Print a counter: knock #1, #2, … until SecondActivity stays open.

B) Clean bypass (one shot)

Hook MainActivity.onResume() to get a live Activity instance. Construct an explicit internal Intent to SecondActivity with from_password=true. Call startActivity(intent) on the main/UI thread.

SecondActivity is not exported; starting it from our own host process (via Frida) is legit as long as we call it from the app's context.

4) Frida snippets

A) "Knock" loop

Java.perform(() => {
  const Activity = Java.use('android.app.Activity');
  const Intent   = Java.use('android.content.Intent');
  const Main     = Java.use('com.hacking.test.MainActivity');

  let knock = 0;

  Main.onResume.overload().implementation = function () {
    this.onResume(); // call original
    knock++;
    console.log('🔔 knock #' + knock);

    Java.scheduleOnMainThread(() => {
      // Re-open MainActivity to trigger SecondActivity's counting path
      const i = Intent.$new(this.getApplicationContext(), Main.class);
      this.startActivity(i);
      this.finish(); // close current to accelerate the loop
    });
  };
});

B) Direct navigation (clean path)

Java.perform(() => {
  const Activity = Java.use('android.app.Activity');
  const Intent   = Java.use('android.content.Intent');
  const Second   = Java.use('com.hacking.test.SecondActivity');

  // Hook any reliable lifecycle point to capture a valid context
  const Main = Java.use('com.hacking.test.MainActivity');

  Main.onResume.overload().implementation = function () {
    this.onResume(); // original

    Java.scheduleOnMainThread(() => {
      try {
        const act = Java.cast(this, Activity);
        const i = Intent.$new(act, Second.class);
        i.putExtra('from_password', true);
        console.log('[+] onResume: ' + act.getClass().getName() + ' — starting SecondActivity');
        act.startActivity(i);
      } catch (e) {
        console.log('[!] startActivity failed: ' + e);
      }
    });
  };
});

5) The flag (mirrored UI)

The hidden layout draws the string mirrored (rotated/inverted). Two easy ways to read it:

  • Visual: take a screenshot and flip horizontally; or
  • Programmatic: hook android.widget.TextView.setText(CharSequence) and print the string argument.

6) Takeaways

  • Client-side checks (random passwords, counters) do not protect secrets.
  • If a screen contains sensitive info, server-side authorization must gate it; don't rely on intent extras or local flags.
  • Rendering tricks (mirroring/rotating) are trivial to bypass with instrumentation.
